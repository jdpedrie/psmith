# Native client architecture (macOS + iOS)

Companion to [docs/architecture.md](../architecture.md) (server side) and [docs/clients/screens.md](screens.md) (per-screen UI/functionality inventory). This doc covers how the macOS and iOS apps are structured, what code they share, and the per-platform native affordances each gets.

## TL;DR

Two app targets, one shared Swift Package. About 70-80% of non-UI code (networking, domain types, state management) is shared as plain Swift. UI atomics are shared too; composite screens and root navigation are platform-specific. The iOS app is genuinely thin because Clark's whole reason for existing is server-side stream resilience — the iOS client just subscribes, persists `lastSeen`, and re-subscribes on foreground.

## Repository / target layout

```
ClarkSwift/                       Swift Package (multi-platform)
  Package.swift
  Sources/
    ClarkKit/                     Pure logic, no UI — works on macOS + iOS
      Generated/                  connect-swift codegen from clark/v1/*.proto
      Auth/                       Bearer token storage (Keychain on both platforms)
      Repository/                 High-level wrappers around ConnectRPC clients
      StreamSubscriber/           AsyncSequence wrapper around SubscribeStream
      LocalCache/                 SwiftData (or plain Codable) for offline browsing
      Plugins/                    Client-side plugin metadata helpers (rendering ConfigSchema forms, etc.)
    ClarkUI/                      Shared SwiftUI views
      Atomic/                     MessageBubble, ChunkRenderer, ChoicesPicker, PluginConfigForm, …
      Composite/                  ConversationView, ProfileEditor — share where reasonable, peel off `#if os` only when the experience really diverges
clarkd-mac/                       macOS app target
  ClarkMacApp.swift               @main entry, root window group, menu commands, Spotlight indexing
  Windows/                        Per-conversation windows, Preferences, Stream Runs panel
  Services/                       NSServices entries, Spotlight delegates
clarkd-ios/                       iOS app target
  ClarkIOSApp.swift               @main entry, tab/sidebar root, scene phase handling, push registration
  Background/                     URLSession background tasks, BGProcessingTask handlers, Notification Service Extension
```

`ClarkKit` and `ClarkUI` are products of the same Swift Package; both app targets list both as dependencies. Apps can also import platform-specific frameworks (`AppKit`, `UIKit`) freely — those aren't allowed inside the package.

## Layer responsibilities

### ClarkKit (cross-platform, no UI)

- **Generated:** Swift bindings for every `.proto` Clark exposes, generated by [connect-swift](https://github.com/connectrpc/connect-swift). Streaming RPCs surface as `AsyncThrowingStream` so they compose with `.task` and `for-await`.
- **Auth:** stores the bearer token in Keychain (`kSecAttrAccessibleAfterFirstUnlock`), provides a Connect interceptor that adds `Authorization: Bearer …` to every request, and surfaces 401 → "needs re-auth" via a published signal.
- **Repository:** high-level idiomatic Swift wrappers around the generated clients — turns "build a request, send, unwrap response.Msg" into Swift methods like `repo.profiles.list()`, `repo.conversations.send(text:on:)`, `repo.streams.subscribe(runID:from:)`. Every UI view talks only to repositories, never to generated clients directly.
- **StreamSubscriber:** the resilience workhorse. Wraps `SubscribeStream` in an `AsyncSequence` that survives transient disconnects: on connection loss, re-subscribes with `from_sequence = lastSeen + 1`. Persists `lastSeen` per stream_run to UserDefaults / SwiftData so a cold app launch can resume.
- **LocalCache:** SwiftData (preferred for new builds) or plain Codable JSON files for: conversations list, recent contexts, recent message chains. Read-only mirror of server state — never the source of truth, never reconciles conflicts.
- **Plugins:** client-side helpers for the chat-plugin system. Renders config forms from `PluginType.config_schema` (JSON Schema), validates user input against that schema, and surfaces capability badges from `PluginCapabilities`.

Estimated split: 70-80% of all non-UI code lives here. Both apps consume the exact same `ConversationRepository`, `StreamSubscriber`, `PluginManager`. Bug-fixing the wire layer means fixing it once.

### ClarkUI (shared SwiftUI)

The trickier shared layer. SwiftUI is multi-platform but diverges on navigation, input affordances, and common chrome (toolbars vs nav bars, popovers vs sheets, hover vs touch). Rule of thumb:

- **Truly atomic views** — `MessageBubble`, `ChunkRenderer` (renders a streaming chunk live), `ChoicesPicker` (the lettered_choices buttons), `PluginConfigForm`, `TokenCountBadge`, `CacheHealthBadge`, `ModelPickerRow`. **Fully shared.** No `#if os` needed.
- **Composite screens** — `ConversationView`, `ProfileEditor`, `ProviderDetail`. **Start shared, peel off `#if os` blocks where the experience really diverges.** A small amount of conditional compilation here is fine; if it grows past a handful of branches, split the screen into per-platform variants importing the shared atomics.
- **Top-level navigation** — `RootView`, `Sidebar`, `TabBar`. **Always per-platform.** Each app target owns its own root.

Don't try to share more than is comfortable. The user feels the platform; pretending to write once and run everywhere produces apps that feel slightly off on both. The cost of two view trees is paid in week-1 effort, never again.

### App targets

Each app target is small — root view, scene/window setup, platform-only chrome, lifecycle handlers. Should be measurable in hundreds of lines, not thousands.

**macOS app target** owns:
- `WindowGroup` setup with `NavigationSplitView` for the list-detail sidebar.
- Multiple-window support (each conversation can detach into its own window via `OpenWindowAction`).
- Menu bar via `MenuBarExtra` and the standard menu commands.
- Spotlight indexing (`CSSearchableIndex`) of conversation titles + recent message bodies.
- NSServices entries for "New Clark conversation with selection."
- Optional menu-bar status item showing active stream count.

**iOS app target** owns:
- `TabView` (iPhone) or `NavigationSplitView` (iPad) at the root.
- Scene-phase handling: on `.background`, persist `lastSeen` per active stream; on `.active`, fire reconnects.
- BGProcessingTask registration for background catch-up.
- Push notification registration + Notification Service Extension to pre-fetch completion message bodies.
- Share extension target for "Send to Clark."

## Why Clark's iOS app is unusually thin

The architecture's defining feature — server-side stream supervision that survives client disconnect — means the iOS app doesn't need to do anything heroic to keep up with long-running model responses. The flow:

1. User sends → server starts `stream_run`, persists chunks as they arrive, fans out to live subscribers.
2. iOS client backgrounds → its `SubscribeStream` connection drops.
3. Server keeps consuming the upstream chunks; supervisor materializes the assistant message when the stream completes; row is in `messages`. Cost is recorded, cache observability is recorded. All without a connected client.
4. iOS foregrounds → re-subscribe with `from_sequence = lastSeen + 1` → server replays missed chunks from `stream_chunks` then live-tails any further (or sends the terminal event if the run finished).
5. Optional: push notification on stream completion → Notification Service Extension pre-fetches the final message body so it's already in cache when the app launches.

There's no need for a complicated client-side state machine or any "what if I miss chunks" logic. The server already solved it. The client is allowed to be naive.

## Auth model

Bearer token in Keychain, set during Login, included on every RPC by the Connect interceptor. 401 surfaces as a `NeedsReAuth` signal that the root view observes; flipping it routes back to Login. Token persists across app launches; revocation happens server-side (admin-deleted session).

Multi-device per user is on the architecture's "Open threads" list; for v1 each device just gets its own session token via a normal login. When per-device pairing lands, the iOS share extension and macOS app could share a paired identity but still separate sessions.

## Local cache posture

Conservative: cache enough to render the conversation list and the most recent context's messages without a network roundtrip on launch. **Don't** cache anything that the server might mutate: cost numbers (recomputed at materialization), cache observability metrics (computed per-turn on the server), token counts (call `CountContextTokens` on demand). The server is the source of truth; the cache is a presentation accelerator.

No write-side cache. Every mutation goes through the server. No queued sends, no reconciliation logic, no merge conflicts. If you're offline, you can browse the cached read-only view; you can't compose. (See "Open question" below for whether to revisit.)

## Build estimate

Bottom-up:
- Wiring `ClarkKit` (auth, repositories, StreamSubscriber, local cache scaffolding) — **~3 days**.
- Single-conversation MVP (login → list → conversation → send → subscribe → render) on both platforms sharing ClarkKit — **~1 week**.
- Profile + plugin + provider/model management screens — **~1 week**.
- Forking, branch navigation, context history, compaction UI — **~3-4 days**.
- macOS-native chrome (multi-window, menu, Spotlight, services) — **~1 week**.
- iOS-native chrome (background tasks, push, share extension) — **~3-4 days**.

Realistic total to a polished pair: ~3-4 weeks of focused work for both platforms.

## The main tradeoff

Single codebase + heavy `#if os(...)` branching saves lines now and pays forever in conditional-compilation chaos. Two app targets with a shared kit means maintaining two view trees, but each feels native and bug-fixes localize. We pick the latter. Clark's value is mostly in the backend (stream resilience, plugin pipeline, compression, history-builder); the apps are the presentation layer. Investing in feel beats saving setup time.

## Open question

**Offline composition.** Should the iOS app let the user write a message while disconnected, then send when reconnected? It's a meaningful design ripple — needs a local outbox + retry/resume semantics, plus a richer local cache (the message tree must be locally addressable). "Online only" is much simpler and is probably acceptable for personal use (the device is online ~all the time anyway). Defer until a real use case surfaces.

## Library / SDK decisions

- **connect-swift** for the RPC layer. Generates Swift clients from the `.proto` files Clark already has; supports the same Connect/gRPC-Web protocols the server exposes. Streaming surfaces as `AsyncThrowingStream`, which composes naturally with SwiftUI's `.task`.
- **SwiftUI** for the UI on both platforms. UIKit/AppKit only for app-target chrome (windows, menus, Spotlight, push delegates) where SwiftUI can't reach.
- **SwiftData** for the local cache. Codable JSON files are an acceptable simpler fallback; pick SwiftData unless cache invalidation logic gets uncomfortable.
- **No third-party state-management framework** (no TCA, no Redux). Plain `@Observable` view models + repository injection. Clark's app surface isn't large or async-coordinated enough to justify the framework tax.
- **Keychain Services directly** for token storage. No third-party wrapper.
